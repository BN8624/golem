# Atelier 결정 노트

## 왜 캐논 채점기부터인가

골렘의 모든 힘은 exact-match oracle에서 나온다. 소설엔 미학 골든이 없으므로, oracle을 **캐논 층**
(연속성·세계규칙·인물사실·복선·타임라인)과 **미학 층**으로 쪼갰다. 캐논 층만 기계 채점 가능하고,
실제 작가가 가장 많이 틀리며, "아이디어를 키울 때" 일관성이 무너지는 지점이다. 그래서 `auto_oracle.py`를
캐논 채점기로 먼저 포팅했다 — 골렘의 "굴러가나 → 잴 수 있나" 전환을 소설에서 재현하는 최소 조각.

## 캐논/미학 경계는 미정 (사용자 결정)

사용자가 "아직 모르겠음"을 골랐다. 그래서 채점기는 **캐논만 pass/fail** 하고 미학은 손대지 않게 짰다.
경계는 `canon_check.py`를 실데이터로 돌려 "뭐가 기계로 잡히나"를 본 뒤 데이터로 긋는다.

## 채점 설계

- 골든은 *심는다* — 모순을 일부러 넣은 초고를 만들고 어느 규칙(C1·C2)을 어겼는지 미리 안다.
  그래야 "채점기가 잡는가"를 precision/recall로 잴 수 있다(빌드 합의가 아니라 심은 골든이 기준).
- exact = 누락(fn)·오탐(fp) 둘 다 0. 오탐을 따로 재는 건 채점기가 *없는 위반을 지어내는* 실패가
  캐논 채점에서 누락만큼 치명적이기 때문이다(작가에게 거짓 경보를 주면 신뢰를 잃는다).
- 규칙별 recall로 "어떤 종류 캐논이 놓치기 쉬운가"를 분리한다(auto_oracle의 key_accuracy_by_name 대응).

## 첫 실런 결과 (2026-06-18, canon-20260618-145250)

exact 1.0 / 2-of-2 완전정확 / 안정 1.0 / 오탐 0. **단, 이건 배선과 쉬운 케이스가 된다는 증거일 뿐**이다.
픽스처가 노골적(양손↔왼팔없음, 낮↔밤마법, 4규칙)이라 1.0은 당연에 가깝다. 골렘 G39~G64처럼
**어려운 케이스로 밀어 한계를 깨야** 진짜 신호다 — 다음 픽스처: (a) 미묘한 타임라인 드리프트,
(b) 인물 지식상태 위반(아직 모를 정보를 앎), (c) 직접 인용이 아닌 패러프레이즈된 모순,
(d) 장거리(여러 챕터 누적) 캐논. 여기서 깨지는 지점이 캐논/미학 경계의 실거리다.

## 자족화 — 골렘 폴더 비의존 (2026-06-18)

atelier는 나중에 독립 프로젝트로 *다른 폴더로 이전*한다. 지금 골렘 밑에 둔 건 골렘이 크는 동안
거기서 나온 아이디어(auto_oracle 등)를 바로 차용하려는 임시 위치일 뿐. 그래서 코드 의존을 끊었다.

- golem이 한 방식 그대로(README "config/llm은 복사 사본, 여기가 정본") 인프라를 `lib/`에 vendoring.
  닫힌 의존: config ← (llm, key_usage), + jsonutil(planning._extract_json 발췌). 전부 stdlib+config만.
- `lib/config.py` PROJECT_ROOT = `parent.parent`(=atelier 루트). .env·runs/key_usage.json이 atelier에
  놓여 폴더째 이동해도 따라온다. golem 경로를 다 뺀 격리 import로 이동성 증명.
- canon_check은 `sys.path`에 `lib`만 넣고 golem 경로 0.

## 모델 정직화 — generate 역할 교정

물려받은 auto_oracle 패턴은 `generate("generator")`를 불렀는데 기본값이 26B('손')다(31B는 'critic'=
'머리'). .env의 GENERATOR_MODEL이 주석이라 **첫 1.0 런은 31B가 아니라 26B가 낸 것**이었다. print의
"gemma-4-31b-it"도 실제 호출 모델이 아닌 박힌 상수였다. 교정: `ROLE="critic"`(31B)로 호출하고,
print/summary는 `get_model(ROLE)`로 실제 모델 ID를 낸다. (golem auto_oracle에도 같은 불일치가 있으나
사용자 영역이라 건드리지 않고 짚어만 둠.) → 31B 실수치는 재실행 필요.

**31B 핀 가드 추가 (2026-06-18).** 26B 사용처 전수조사 결과 호출 0(세 스크립트 다 `ROLE="critic"`→31B,
fallback도 critic-ward, 26B 문자열은 lib의 비활성 정의뿐). 다만 canon_check·design_check은 planning과
달리 모델을 명시 핀 안 하고 기본값(31B)에 의존했다 → `.env` 오염이나 `generate("generator")` 추가 시
26B가 샐 잠복 위험. 두 채점기 실콜 경로에 planning과 같은 핀(`GENERATOR_MODEL`·`CRITIC_MODEL`=31B)을
박고 `ROLE!=critic`이면 죽는 가드를 넣었다. replay(키0) 경로는 무영향. 실콜 `--n 1`로 31B 찍힘 확인.

## 캐논/미학 경계 1차 데이터 — 검출↔precision 방향 비대칭 (2026-06-18)

`fixtures_ko_hard`(함의 위반 hard + 위반0 함정 trap)로 채점기를 밀어 첫 한계를 노출했다.

- **검출(recall)은 한국어 함의로도 강함**: hard의 C3(왕가 피 없는 용병이 조각 활성화)·C6(발타자르
  혈관에 선왕 피)를 직접 진술 없이 추론으로 3/3 검출(1.0). 표면 매칭이 아니라 의미 추론을 한다.
- **precision(일치 방향 판정)은 약함**: trap의 "한 어머니에게서 난 친누이"는 C2(친여동생)와 **명백히 일치**
  하는데 31B가 매번(3/3) C2 위반으로 뒤집는다. 즉 "틀린 걸 찾기"는 되는데 "맞는 걸 맞다고 두기"가 안 됨.
- **프롬프트로 불가역**: (1)명시 지시("일치=위반아님, 불확실하면 빈배열") 실패 → (2)양성/음성 few-shot
  (거의 동형인 "same mother sister→NO violation" 예시 포함) 실패. 둘 다 trap 오탐 1.0 그대로.
- **경계 1차 결론**: canon/미학 경계는 "기계로 *검출*되나"가 아니라 "기계가 *거짓 경보*를 억제하나"에 있다.
  검출은 기계화 가능, precision(특히 한국어 일치 문장 방향)이 실한계. 다음 카드는 2패스 검증(콜2배,
  아키텍처 변경) — 별도 세션 안건. 산출 원장: `runs/canon-20260618-170505`(few-shot 최종)·`-165458`·`-164845`.

## 2패스 검증이 precision 한계를 깸 — 경계 재이동 (2026-06-18, canon-20260618-171503)

위 "프롬프트 불가역" 결론을 **아키텍처로 뒤집음**. `--verify`(2패스: 1패스 위반 후보를 별도 콜로
"정말 모순이냐" 재확인 → confirmed만 남김) 적용 결과:
- trap 오탐 1.0 → **0** (C2 "친누이" 후보를 2패스가 reject), hard 검출 C3·C6 **1.0 유지**(진짜 위반은 confirm).
- 전체 exact 0.5 → **1.0** (2/2 완전정확, 안정 1.0, 오탐 0).

**해석**: precision 한계는 *모델 능력*이 아니라 *단일 콜 태스크 프레이밍*의 한계였다. 같은 31B가 "위반을
찾아라"(1패스)에선 과탐지하고 "이게 진짜 모순이냐"(2패스)에선 같은 문장을 옳게 일치=무위반 판정한다.
→ **경계 재이동: precision도 기계화 가능하다, 단 2패스 비용(후보 있을 때 콜 약 2배)을 내면.** 검출/검증을
분리하는 게 캐논 채점의 핵심 패턴. (구현: `_verify`, `--verify` 플래그. 검증 콜 실패 시 1패스 보존 = 보수적.)

## 2패스 스트레스 — 과교정 없음 확인 + 통사적 부정 한계 발견 (2026-06-18, hard2/hard3)

2패스의 두 주장을 별도 픽스처로 스트레스. (a) 1패스 오탐을 거른다, (b) 진짜 위반은 안 깎는다.

- **hard2(canon-180526/180932)**: 측정 가치 0이 됐다. subtle_violation을 직접 진술("두 팔 늘어뜨림",
  "옛 왕가 핏줄이 흐르고")로 둬서 1패스가 이미 완벽(exact 1.0, 오탐 0) → 2패스가 깎을 것도 거를 것도 없음.
  교훈: 위반이 표면 진술이면 31B가 1패스에서 다 잡아 2패스 차이를 못 만든다. 위반은 **함의에 숨겨야** 측정된다.
- **hard3(canon-185805 1패스 / 190219 2패스)**: 위반을 행동·정황에만 숨기고(왼손으로 활대 받침=양손 추론
  C1, 비왕가 아이 손의 빛=C3), clean 함정에 통사적 덫을 깔았다("친누이가 *아니*라는 헛소문을 일축"=이중부정).
  - **(b) 과교정 없음 — 재확인**: subtle_violation 검출 C1·C3가 1패스 1.0 → 2패스 **1.0 유지**. 미묘한
    진짜 위반도 2패스가 안 깎는다. precision↔recall 트레이드오프는 안 나옴 → 2패스 robust가 hard1 단일 사례를 벗어남.
  - **(a) 새 한계 — 통사적 부정**: subtle_clean 오탐 1패스 1.0 → 2패스 **0.33**(잔존, 안정 0.67). hard1
    trap("친누이")은 →0 완전 제거였는데, **이중부정**("X가 아니라는 헛소문을 일축" = 실제론 X 맞음)은 31B가
    부정 구조를 못 풀어 2패스 재확인도 시드마다 흔들린다(3번 중 1번 여전히 오판). 오탐 근거는 정확히 그 한 줄뿐
    (다른 함정 "참칭"·"남은 오른팔"은 안 속음).

**경계 갱신**: 2패스는 *의미 모순*엔 robust하나 *통사적 부정 구조*엔 부분적으로만 강하다. 의미 오탐(친누이=C2
일치)은 검출/검증 분리로 풀리지만, 통사 오탐(부정문 방향 오독)은 2패스로도 안 사라진다 — 별개의 한계 축이다.
→ 다음 카드 후보: 부정/이중부정 정규화(전처리)나 검증 프롬프트에 "부정문 방향" 명시. 단 이건 별도 안건.

## 한국어 synth 빈 패킷 — 재현 안 됨 (2026-06-18)

핸드오프는 "한국어 synthesize = 빈 패킷(OPEN) → 잘림 의심"이라 단정했으나, **코드 수정 없이 재현 실패**.
동일 (draft,issues)로 synth만 11키 병렬 측정 → **빈 패킷 0/11**(단발 1회 + 병렬 11회 = 12/12 FROZEN,
canon 5~7·decisions 8~9). 잘림이 간헐이라면 11번 중 한 번은 났어야 하는데 0이다.

- 판정: planning.py synth엔 토큰 상한 손댄 흔적 없음 → 코드 결함이 아니라 핸드오프 작성 시점의 모델/서버
  일시 상태였다고 본다(이번 런도 5xx가 떴다 재시도 복구 — 그날 서버 불안정 정황).
- 조치: 근본 수정(max_output_tokens 상향) **불필요**. 진단 덤프(`RealCaller.synth`→`runs/_synth_raw.txt`)는
  **제거하지 않고 안전망으로 유지** — 파싱 실패시만 작동하고 평소 무해. 0/11이 영구 무결 보장은 아니므로.

## design 단계 출범 — setup→payoff 채점기 (2026-06-18, design-20260618-192236)

frontier 3. canon_check이 양방향(거름/안 깎음)으로 닫혀 design을 얹었다. 설계 결정:

- **채점기 먼저**(생산자 뒤로). canon_check가 채점기 1개로 먼저 태어나고 planning(생산자)이 뒤따른
  선례 그대로. 측정 가치는 채점기에 있다 — "31B가 setup→payoff 실패를 짚나"가 frontier 질문이다.
- **canon_check의 정확한 거울**. canon_check은 "동결 *사실*을 *모순*하나"를, design_check은 "동결
  *약속*을 *회수*하나"를 잰다. 방향만 반대(모순 검출 ↔ 미회수 검출)라 검출/검증 2패스 + exact/recall/
  오탐/안정성 기계가 그대로 돈다. `rule_id`→`setup_id`, `violations`→`unresolved`만 바뀜.
- **골든은 심는다**(canon과 동일). 손으로 쓴 비트시트 2벌 — 클린(설정 S1~S5 전부 회수) + 미회수
  (S2 펜던트·S4 부대몰살을 *언급은 하되 회수만 안 함*). 골든=[S2,S4]. 노골적 부재가 아니라 "도입됐는데
  payoff 비트가 없음"이라야 진짜 frontier다(쉬운 픽스처 경계 = canon hard1과 같은 함정 설계).
- 픽스처 연속성: 기존 FROZEN 한국어 바이블(카엘·리아 남매)을 비트시트로 이어 같은 세계를 쓴다.
- replay_demo는 일부러 시드를 흔들어(클린 seed2가 S2 헛잡음, 미회수 seed2가 S4 놓침) exact≠1.0이
  나오게 심었다 — 채점 수학이 *실제로 구분*하는지 키 0으로 증명(canon replay_demo와 같은 의도).
- 키 0 검증 통과: clean exact 0.67/오탐 0.33, unresolved exact 0.67/recall S2 1.0·S4 0.667, `[S4]`→`S4`
  정규화 작동. 수학 전부 예측과 일치. **★실콜은 사용자 go 뒤.**

## design 채점기 ★실런 + 생산자 design.py (2026-06-18)

- **design_check ★실런**(design-193115 1패스 / 193408 2패스): exact 1.0, clean 오탐 0, unresolved
  S2·S4 recall 1.0, 2패스도 1.0 유지. **과교정 없음** 재확인했으나, 1패스가 이미 완벽해 2패스 *차이*는
  측정 못 함 — canon hard2와 같은 상황(미회수를 너무 또렷이 둠). 2패스 가치를 재려면 회수를 정황에만
  숨기거나 회수처럼 보이는 함정을 깐 어려운 픽스처가 필요(다음 카드).

## 외부 리뷰 반영 — FROZEN 게이트 결함 등 수정 (2026-06-18)

외부 리뷰(GPT/Codex)가 코드를 읽고 짚은 결함을 검증·수정. 우선순위 1·2·5 묶음.

- **결함(1) FROZEN 게이트 거짓 — 실재 버그였다.** `resolved = bool(decisions or assumed or deferred)`는
  셋 중 하나만 비지 않으면 True라, BLOCKING 5개 중 1개만 답해도 FROZEN. STATUS의 `미해소 BLOCKING:

  `resolved = ... and unresolved_block==0`. STATUS는 실제 미해소 수를 찍는다. **완전한 1:1 해소 추적은
  모델이 자유서술로 답해 자동 매핑 불가(LLM 판정 필요) → 별도. 여기선 "질문보다 답이 적으면 OPEN"만 보장**

- **결함(2) ID 무결성** — canon/setup ID 중복(C1 두 개 등)을 안 잡았다. `dup_ids` 검사 추가, frozen 조건에

- **결함(5) 단일파일 덮어쓰기** — planning A/B/C 결과가 `planning_result.json`에 매번 덮여 재현성 약함.
  `planning-{stamp}.json`(영구 보존) + `planning_result.json`(최신 포인터) 둘 다 쓰게 변경. canon/design_check은
  이미 타임스탬프 폴더라 무관.
- 검증: `_write_packet` 직접 호출 단위테스트 — BLOCKING3/흡수1→OPEN(unresolved 2), canon ID중복→OPEN,
  정상→FROZEN, design 동일. 실 replay 회귀 둘 다 FROZEN 유지(BLOCKING 2/흡수 5).

  적 없고 "합의 확인"이라 이미 명시했다.

## specQA 단계 출범 — 캐논/미학 격리 채점기 (2026-06-18)

frontier 4 시작. NovelStudioMode의 oracle 2층 분리("검증 가능한 캐논 vs 검증 불가한 미학")를 31B가
기계화할 수 있나를 잰다. canon_check·design_check가 둘 다 *검출* 과제였던 것과 달리 specQA는 *격리(분류)*
과제다 — 이게 이 단계의 본질이라 채점 모양을 여기 못박는다.

**왜 분류 과제인가.** NovelStudioMode 매핑표 specQA 행 = "씬 계약 + 캐논/미학 위험 분리, 검증 불가한
기준 = 미학으로 격리". 즉 씬 계약(기준 목록)을 받아 각 기준을 캐논(검증가능)/미학(검증불가)으로 가르는 게
전부다. "초고를 계약 대비 검증"은 specQA가 아니라 build/integration 범위라 제외했다. 핸드오프가 못박은
frontier 질문("31B가 검증 가능한 씬 기준과 미학을 옳게 가르나")과도 정확히 일치한다.

**채점 모양(design_check 거울).** 입력 = premise + 씬 기준 목록(라벨 없이). 출력 = 31B가 '검증가능
(캐논)'이라 라벨한 기준 집합. 골든 = 심은 캐논 ID 집합. found==golden이면 exact. design_check의
detect/verify 2패스·N시드·exact·recall·fp·stability 기계를 그대로 쓰고 `unresolved→verifiable`,
`setup_id→criterion_id`만 바꿨다.

**위험한 방향을 fp로 둔 이유(핵심 판단).** 미학→캐논 오라벨(fp)이 NovelStudioMode가 금지한 "합의채점
흉내"(검증 못 할 걸 exact로 재려는 것) 그 자체다. 그래서 프롬프트 보수적 기본값을 "검증가능한지 의심되면
미학으로 둔다"로 박고, verify 2패스도 후보를 "정말 객관 YES/NO 오라클이 있나"로 재확인해 fp를 깎게 했다.
반대 방향(캐논→미학 누락=fn)은 "검증 가능한 걸 사람한테 떠넘김"이라 덜 위험하다 — 비대칭이 의도적이다.

**심은 픽스처 `fixtures_specqa/`**: 카엘·리아 세계 연속. scene_mixed(C1·C3·C4·C6=캐논 — 왼팔 연속성·
S1 회수·타임라인·적대자사실 / C2·C5=미학 — 긴장점층·분위기은유, 골든 [C1,C3,C4,C6]) + scene_aesthetic
(전부 미학 — 침묵·산문리듬·톤대비·공감, 골든 []). 후자는 design "clean"(빈 골든)의 거울이되 **반대 실패**를
잰다 — design은 "헛잡지 마라"(아무것도 미회수 아님), specQA는 "검증가능을 지어내지 마라"(전부 미학).

**키 0 검증 통과**(specqa-20260618-214057): `--replay`로 채점 수학 정확 — scene_mixed exact 0.667
(흔든 시드가 C4 누락+C2 헛라벨), C4 recall 0.667·나머지 1.0, fp 0.33. scene_aesthetic exact 0.667
(흔든 시드가 C1 헛라벨), "(전부 미학 계약)" 출력·fp 0.33. 일부러 흔든 시드가 구분돼 수학 신뢰됨.
fp 진단 계측(어떤 미학 기준을 무슨 근거로 캐논이라 헛라벨하나)도 결과 JSON에 기록됨. ★실콜은 사용자 go 뒤.

## specQA ★실콜 — 격리는 강하나 혼합 기준이 경계 (2026-06-18, specqa-214932/215325)

1패스/2패스 둘 다 실콜. **두 런이 완전 동일**: exact 0.5, 안정 1.0, fp 0.

- **금지된 방향이 0(핵심 성공).** scene_aesthetic(전부 미학, 골든 [])에서 31B가 검증가능 기준을 **하나도
  지어내지 않음**(exact 1.0, fp 0, 안정 1.0). NovelStudioMode가 가장 경계한 "검증 못 할 걸 exact로 재려는
  합의채점 흉내"가 실데이터에서 나타나지 않았다. 미학을 미학으로 옳게 격리한다.
- **유일 오류는 안전한 방향의 fn.** scene_mixed에서 C1·C3·C4(왼팔 연속성·S1 회수·타임라인)는 recall 1.0
  으로 캐논 라벨, 그런데 **C6(발타자르=적대자, 우호적 묘사 금지)을 3/3 안정적으로 미학으로 흘렸다.** 즉
  검증가능한 걸 과도하게 미학으로 격리(fn). 위험한 fp가 아니라 보수적 fn 쪽 실패다.
- **C6은 model 실패가 아니라 혼합 기준의 모호성일 수 있다(경계 해석).** C6 = 사실("발타자르는 적대자다",
  검증가능) + 톤("우호적으로 묘사돼선 안 된다", 해석 여지). 모델은 작동 절(묘사가 우호적이냐)을 취향 판단
  으로 읽어 미학으로 분류했다 — 변호 가능한 판정이다. 노골적으로 발타자르를 우방으로 그리면 객관적 캐논
  위반이지만, 경계에선 "우호적이냐"가 흔들린다. **진짜 격리 경계는 *혼합 기준*에 있다** — 잘 쓰인 씬 계약은
  사실 축과 톤 축을 원자화해 따로 적어야 한다(설계 통찰, 다음 hard 픽스처로 검증).
- **2패스 측정가치 0이되 canon·design과 이유가 다르다(비대칭의 새 축).** canon은 2패스가 precision 한계
  (fp)를 깼고, design은 1패스부터 fp·fn 양방향 강했다. specQA는 **verify가 구조적으로 fp(과대주장)만 깎는
  도구라, 오류가 fn(과소주장)인 이 케이스엔 닿지 않는다.** 그래서 2패스=1패스. 과교정도 0(C1/C3/C4 유지)
  이라 verify가 해롭진 않다. 교훈: specQA의 위험 방향(fp)을 깨려면 *fp가 나는* 픽스처(미학을 구체어로
  위장)가 필요하고, fn을 줄이려면 verify가 아니라 *detect 프롬프트*나 계약 원자화가 답이다.

## specQA hard ★실콜 — 함정 안 먹힘, C6 가설 확정 (2026-06-18, specqa-220612/220910)

- **fp 함정이 안 먹혔다.** scene_fp_trap은 미학을 구체어로 위장한 4기준("3단계 점층"·"톤 일관"·"고르게
  배치"·"이음매 없이 전환")이고 골든은 []. 31B는 표면의 검증돼-보이는 단어에 낚이지 않고 **전부 미학으로
  옳게 격리**(fp 0, 안정 1.0). NovelStudioMode가 금지한 "합의채점 흉내"는 미끼를 던져도 나타나지 않는다.

- **C6 가설 확정.** scene_atomized는 순수 원자화 캐논 4기준(발타자르 등장·S2 회수·타임라인 선행·왼팔
  연속성), recall 전부 1.0. → 본 fixture의 C6 미스는 **모델 능력 한계가 아니라 혼합 기준(사실+톤이 한
  기준에 뭉친 것)의 모호성** 탓임이 대조로 확정됐다. 사실만 떼면 31B는 안정적으로 캐논으로 잡는다.
- **2패스 과교정 0.** verify가 원자화 캐논을 하나도 안 깎음(recall 1.0 유지). design과 동형 — 1패스가
  이미 완벽해 2패스 측정가치 0, 무해.
- **정직한 미검증.** 함정이 fp를 못 냈으므로 "2패스가 specQA의 fp를 깎나"(canon precision판)는 **깰 fp가
  안 생겨 구조적으로 측정 못 했다.** design hard1과 같은 상황(모델이 예상보다 강함). 더 센 fp 함정을
  설계할 수 있으나, 위험축이 이미 robust로 나온 이상 한계효용은 낮다.

**specQA 잠정 결론(경계 종합).** 캐논/미학 격리는 31B **1패스로 충분**하다 — 순수 캐논 recall 1.0, 순수
미학 fp 0(미끼에도 0). canon(모순검출)이 precision에 2패스를 요한 것과 비대칭이고, design(setup→payoff)

문제가 아니라 **씬 계약을 원자화해 사실 축/톤 축을 분리하라는 생산자(producer) 몫**이다. → 다음 단계는
